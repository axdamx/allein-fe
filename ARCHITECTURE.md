## 1. Overview

Allein is a full-stack SaaS platform for building, deploying, and managing AI agents. Users create agents from 6 pre-configured types (property, insurance, car dealer, travel, sales, legal), chat with them, and let agents perform actions (create leads, set reminders) using the user's uploaded knowledge base (RAG).

### Core capabilities
- **Config-driven agents**: 6 agent types, each with custom system prompts and tools
- **Real AI chat**: GLM-4.5-Flash via Vercel AI SDK with native tool calling
- **RAG knowledge base**: Upload documents → automatic chunking + embedding → semantic retrieval at chat time
- **CRM pipeline**: Leads, deals (kanban), reminders, pipeline tracking
- **Tier gating**: 4 plans (Free/Lite/Pro/Custom) with dual enforcement (UI + server)
- **SSR**: Full server-side rendering via TanStack Start

## 4. Database Schema

### Core tables

```
profiles (auth.users 1:1)
├── id (uuid, FK → auth.users)
├── email, full_name, avatar_url, company, phone
├── plan (enum: free | lite | pro | custom)
├── role (enum: member | admin | owner)
├── agents_count, conversations_count, messages_count  ← usage counters
├── posts_count, documents_count
└── trial_ends_at, subscription_id

agent_types (catalog, seeded)
├── key (enum: property | insurance | car_dealer | travel | sales | legal)
├── label, description, icon (lucide name, kebab-case)
├── system_prompt (base prompt per type)
├── default_tools (jsonb)
└── sidebar_items (jsonb — custom nav per type)

agents (user instances)
├── id, owner_id, type (FK → agent_types)
├── name, status (active | paused | draft | archived)
├── system_prompt (override of type default, nullable)
├── model (default: glm-4.5-flash)
└── conversations_count, leads_count
```

### Chat tables
```
conversations
├── id, agent_id, owner_id
├── title (auto-generated from first message)
├── is_archived
└── created_at, updated_at

messages
├── id, conversation_id
├── role (user | assistant | system | tool)
├── content, model
└── tokens_in, tokens_out
```

### CRM tables
```
leads
├── id, owner_id, agent_id (nullable)
├── name, email, phone, company
├── source (website | whatsapp | phone | email | social | referral | other)
├── status (new | contacted | qualified | negotiation | won | lost)
├── value, notes, tags[]
└── last_contacted_at

deals
├── id, owner_id, lead_id (nullable)
├── title, stage (lead | qualified | proposal | negotiation | closed_won | closed_lost)
├── value, currency, probability (auto-set by stage)
└── expected_close_date

reminders
├── id, owner_id, lead_id, deal_id
├── title, description, due_at
├── channel (in_app | email | whatsapp)
└── status (pending | sent | snoozed | done)
```

### RAG tables (pgvector)
```
documents
├── id, owner_id, agent_id (nullable = shared)
├── name, mime_type, size_bytes
├── status (processing | extracting | chunking | embedding | storing | ready | failed)
└── chunk_count

document_chunks
├── id, document_id, owner_id
├── content (text), chunk_index
└── embedding (vector(384))  ← HNSW index for cosine similarity

match_documents(query_embedding, match_count, filter_owner_id, filter_agent_id)
  → semantic search via pgvector cosine similarity
```

## 5. Authentication Flow

```
1. User visits / → beforeLoad checks context.user
2. Not logged in → redirect to /login
3. AuthForm → submit → loginFn (server function)
4. loginFn → supabase.auth.signInWithPassword() → sets session cookie
5. Client: router.invalidate() → navigate to /dashboard
6. Dashboard route → _authed guard → context.user must exist
7. Logout → logoutFn → supabase.auth.signOut() → redirect to /login
```

### Key files
- `src/server/auth.ts` — `loginFn`, `signupFn`, `logoutFn`
- `src/lib/supabase/server.server.ts` — SSR client with cookie handling
- `src/routes/__root.tsx` — `beforeLoad` fetches user via `fetchUser()` server fn
- `src/routes/_authed.tsx` — layout guard, redirects to /login if no user

### Important: `.server.ts` convention
Files ending in `.server.ts` are server-only. TanStack Start's import-protection
plugin refuses to bundle them into client code. This prevents server secrets
(Supabase keys, LLM API keys) from leaking to the browser.

```
src/lib/supabase/server.server.ts     ← server-only (has server imports)
src/server/profile.server.ts          ← server-only (imports supabase client)
src/server/profile.ts                 ← shared (RPC wrappers, safe for client)
```

The RPC layer (`profile.ts`) uses dynamic `import('./profile.server')` inside
the handler so the server-only code is loaded only on the server.

## 6. Tier Gating System

### Single source of truth: `src/lib/plans.ts`
```typescript
PLAN_CONFIGS = {
  free:  { limits: { agents: 1, messages: 100, ... }, features: { crm: false, ... } },
  lite:  { limits: { agents: 3, messages: 2000, ... }, features: { crm: true, ... } },
  pro:   { limits: { agents: 10, messages: null (unlimited), ... }, features: { ... } },
  custom:{ limits: { all unlimited }, features: { all true } },

```

### Dual enforcement
| Layer | How | Example |
|---|---|---|
| **Frontend (UX)** | `<FeatureGate>`, `<UsageGate>`, `usePlan().canDo()` | Hides CRM from Free users |
| **Backend (security)** | `enforceLimit('agents')`, `requireFeature('crm')` | Throws error if over limit |

### Components
- `FeatureGate` — wraps children, shows locked screen if feature unavailable
- `UsageLimitBanner` — shows at 80%+ usage, red at 100%
- `UpgradeModal` — tier comparison with context-aware messaging
- `PlanBadge` — colored tier indicator

## 8. AI Chat with RAG

### The complete chat flow (Vercel AI SDK)

```
User types message
    ↓
useChatStream.send(content)
    ↓
1. Optimistic update — user message appears instantly (TanStack Query setQueryData)
    ↓
2. sendMessage() RPC → server function
    ↓
3. sendMessageImpl():
   a. Load conversation + agent config from DB
   b. Load conversation history (last 20 messages)
   c. Save user message to DB
   d. RAG: retrieveContext() → embed query → pgvector search → top 5 chunks
   e. Build system prompt: agent prompt + RAG context + tool instructions
   f. generateText({ model, messages, tools: agentTools, stopWhen: stepCountIs(3) })
      ↓
      The LLM can:
      - Answer directly (text response)
      - Call createLead({name, email, phone}) — native tool calling
      - Call createReminder({title, due_at})
      - Multi-step: call tool → see result → respond
      ↓
   g. Collect tool results from result.steps
   h. Save assistant message to DB
   i. Auto-generate title (first message only)
    ↓
4. Return { reply, toolCalls } to client
    ↓
5. Client reveals reply progressively (typing animation, ~12ms per chunk)
    ↓
6. Tool results → toast notification + query invalidation (leads show instantly)
```

### RAG pipeline (document upload)

```
Upload .txt/.md/.pdf
    ↓
1. Client reads file (text for .txt, base64 for .pdf)
    ↓
2. uploadDocumentImpl():
   a. Create document row (status: processing)
   b. Status: extracting → pdf-parse (PDFs) or raw text
   c. Status: chunking → chunkText() splits by paragraphs (500 chars, 50 overlap)
   d. Status: embedding → embedBatch() via Transformers.js (384-dim, local)
   e. Status: storing → insert chunks + vectors into document_chunks
   f. Status: ready
    ↓
3. UI polls every 1.5s while processing (shows progress bar)
    ↓
4. At chat time: retrieveContext() embeds the query → match_documents RPC
   → cosine similarity → chunks above 12% threshold → injected into system prompt
```

### Why local embeddings (not API)?
- **Free forever** — no OpenAI/ZAI API cost
- **384 dimensions** (smaller than OpenAI's 1536) → faster search, less storage
- **Runs on CPU** via Transformers.js (Xenova/all-MiniLM-L6-v2, ~25MB)
- **Tradeoff:** lower similarity scores (threshold lowered to 0.12 vs 0.3)

## 10. Key Patterns

### Pattern: Server function RPC layer
```typescript
// profile.ts (shared, safe for client import)
export const getPlanState = createServerFn({ method: 'GET' }).handler(
  async () => {
    const { getPlanStateImpl } = await import('./profile.server')
    return getPlanStateImpl()
  },
)
```
- The `.ts` file has NO server-only imports at module scope
- The handler dynamically imports the `.server.ts` implementation
- TanStack Start compiles this into an RPC endpoint automatically

### Pattern: Optimistic updates
```typescript
// Show user message instantly before server responds
qc.setQueryData<MessageRow[]>(
  ['chat', 'messages', conversationId],
  (old) => [...(old ?? []), optimisticMessage],
)
```

### Pattern: Native tool calling (Vercel AI SDK)
```typescript
const createLeadTool = tool({
  description: 'Save a contact as a lead...',
  inputSchema: z.object({ name: z.string(), email: z.string(), ... }),
  execute: async ({ name, email }) => {
    // Actually insert into Supabase
    return { success: true, message: 'Lead created' }
  },
})

// LLM calls this automatically with guaranteed-type params
const result = await generateText({
  model, messages, tools: { createLead: createLeadTool },
  stopWhen: stepCountIs(3),  // allow tool → result → respond
})
```

### Pattern: Polling for async operations
```typescript
// useDocuments polls every 1.5s while any doc is processing
refetchInterval: (query) => {
  const hasProcessing = query.state.data?.some(d => d.status === 'processing')
  return hasProcessing ? 1500 : false
}
```

### Pattern: RLS (Row Level Security)
```sql
-- Every table has owner-based policies
create policy "leads_select" on public.leads
  for select using (owner_id = auth.uid() or public.is_admin());
create policy "leads_insert" on public.leads
  for insert with check (owner_id = auth.uid());
```
